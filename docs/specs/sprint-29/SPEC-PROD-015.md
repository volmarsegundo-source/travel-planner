# SPEC-PROD-015: Trip Duplication from Summary

**Version**: 1.0.0
**Status**: Draft
**Author**: product-owner
**Reviewers**: [tech-lead, architect]
**Created**: 2026-03-12
**Last Updated**: 2026-03-12

---

## 1. Problem Statement

Travelers frequently return to the same destinations -- annual family vacations, repeat business trips, or revisiting a favorite city. When a traveler wants to plan a return trip to a destination they have already visited, they must start from scratch: selecting the same destination, re-entering travel style preferences, and re-configuring passenger details. This repetitive work negates the "planning intelligence" that Atlas accumulates during the first trip.

A "Plan a similar trip" action on the summary page of a completed trip allows travelers to bootstrap a new trip with pre-populated data from a previous trip, reducing planning friction for repeat destinations from minutes to seconds.

**Evidence**: Travel industry data (Phocuswire, 2024) indicates that 35-40% of leisure travelers revisit a destination within 3 years. Corporate travelers have even higher repeat rates (60-70%).

**Affected users**: All personas. Primarily @leisure-family (annual vacations) and @business-traveler (repeat business destinations).

## 2. User Story

As a traveler who has completed a trip to a destination I enjoyed,
I want to quickly create a new trip to the same destination with my previous preferences and passenger configuration pre-filled,
so that I can start planning a return visit without re-entering information I have already provided.

### Traveler Context

- **Pain point**: Starting a trip from scratch when you have already planned a similar one feels wasteful. The product knows my destination preferences, travel style, and passenger configuration -- why do I need to re-enter them?
- **Current workaround**: Manually create a new trip and re-enter all details from memory or by referencing the old trip.
- **Frequency**: 1-2 times per year for leisure travelers with repeat destinations. More frequent for business travelers.

## 3. Acceptance Criteria

### Duplication Action

- [ ] AC-001: The summary page for completed trips (expedition status = "completed") MUST display a "Plan a similar trip" / "Planejar viagem similar" action button.
- [ ] AC-002: The button MUST NOT appear for trips that are in progress (expedition not completed). Travelers should complete their current trip before duplicating it.
- [ ] AC-003: The button MUST follow the CTA hierarchy defined in SPEC-PROD-009: tertiary visual prominence (below "Edit" actions, below "Share" action).

### Data Copied

- [ ] AC-004: The duplicated trip MUST be pre-populated with: (a) destination (including destinationLat/destinationLon coordinates), (b) travel style preferences from Phase 2, (c) passenger count (adults, children, infants, seniors).
- [ ] AC-005: The duplicated trip MUST have blank/default values for: (a) dates (start date, end date -- traveler must set new dates), (b) transport bookings (no booking codes, carrier names, or segment data copied), (c) accommodation bookings (no hotel names, booking codes, or addresses copied), (d) mobility selections (reset to default), (e) checklist items (will be regenerated by AI for the new trip), (f) itinerary (will be regenerated by AI for the new trip), (g) budget entries (new trip starts with zero expenses).
- [ ] AC-006: The trip origin field MUST be copied from the original trip (the traveler likely departs from the same city).

### Trip Creation Flow

- [ ] AC-007: After clicking "Plan a similar trip", the system MUST create a new Trip record with the pre-populated data and redirect the user to Phase 1 of the new trip. The destination field is pre-filled but editable (the traveler may want to change it).
- [ ] AC-008: The new trip MUST appear in the Expeditions list immediately.
- [ ] AC-009: Gamification points MUST be awarded for creating the new trip (using existing PointsEngine logic for trip creation). No bonus points for duplication specifically.
- [ ] AC-010: The new trip MUST respect the max 20 active trips limit (US-001 AC-007). If the user is at the limit, display an appropriate error message instead of creating the trip.

### Localization

- [ ] AC-011: All new text introduced by this spec MUST be localized for PT-BR and EN.

## 4. Scope

### In Scope

- "Plan a similar trip" button on completed trip summaries
- Pre-population of destination, travel style, passenger count, and origin
- New trip creation with blank dates, no bookings, no checklist, no itinerary
- Redirect to Phase 1 of the new trip
- Gamification points for trip creation
- Max trip limit enforcement

### Out of Scope

- Trip template system (saving a trip as a reusable template -- future)
- Copying checklist items or itinerary from the original trip
- Copying booking data (transport, accommodation) from the original trip
- Bulk trip duplication (duplicating multiple trips at once)
- "Trip series" concept (linking original and duplicated trips)
- Duplication from in-progress trips (must be completed first)

## 5. Constraints (MANDATORY)

### Security

- BOLA: only the trip owner can duplicate their own trip. The duplication action MUST verify trip ownership via authenticated session.
- The new trip MUST be owned by the authenticated user (no transfer of ownership).
- No sensitive data from the original trip (booking codes, passenger PII beyond count) is copied to the new trip.

### Accessibility

- WCAG 2.1 AA compliance.
- The "Plan a similar trip" button MUST have an accessible label including the trip name (e.g., "Plan a similar trip to Paris").
- Error message for max trip limit MUST be announced to screen readers.

### Performance

- Trip duplication (database insert + redirect) MUST complete in under 1 second.

### Architectural Boundaries

- This spec does NOT define the database operation for trip duplication. That belongs to SPEC-ARCH.
- The duplication creates a new, independent Trip record. There is no foreign key relationship between the original and the duplicate.

## 6. Success Metrics

- **Adoption**: >= 10% of users with completed trips use "Plan a similar trip" at least once during beta. Measured by analytics.
- **Completion rate**: >= 60% of duplicated trips reach at least Phase 3 (indicating the pre-populated data reduced friction enough to continue planning). Measured by analytics.
- **Time to Phase 1 completion**: Duplicated trips complete Phase 1 in < 30 seconds (vs. 2-3 minutes for new trips). Measured by analytics.

## 7. Dependencies

- SPEC-PROD-011 (Summary with Edit): The duplication button lives on the summary page.
- Trip model must support the fields being copied (destination, destinationLat, destinationLon, origin -- all exist post Sprint 21+).
- PointsEngine (Sprint 9): existing trip creation points logic.
- Max trip limit enforcement (US-001 AC-007): existing validation.

## 8. Vendor Independence

- This spec describes WHAT the feature does, not HOW it is implemented.
- Must NOT reference specific libraries, frameworks, or vendor products.
- Implementation details belong in the corresponding SPEC-ARCH document.
