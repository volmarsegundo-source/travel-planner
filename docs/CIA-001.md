## Executive Summary

US-001 introduces Trip Creation and Management as the foundational entity of the Travel Planner domain. This is a greenfield implementation on a project with no prior releases, no existing users, no deployed API contracts, and no production database — making this a zero-impact change from a backward compatibility perspective. The assessment confirms version 0.1.0 as the appropriate initial release designation, establishes the semantic versioning baseline, and defines the rollback, deployment, and go/no-go criteria for the first production deployment.

## Breaking Change Analysis

This is a greenfield implementation. The following confirmations apply:

**No existing API contracts are affected.**
The project has no prior deployed REST endpoints and no prior Server Actions. The REST endpoints listed in SPEC-001 Section 4.2 (`GET /api/v1/trips`, `POST /api/v1/trips`, etc.) are explicitly documented as out-of-scope for this spec — they are not implemented and create no contract obligations.

**No existing data is migrated.**
The database contains no user or trip data. The migration creates new tables on an empty schema. There is no data transformation risk.

**No existing users are impacted.**
The system has no production users. There are no clients consuming any API endpoint, no mobile apps pinned to a contract, and no external partner integrations.

**The soft-delete middleware change to `src/server/db/client.ts` is the only modification with broad query scope.** Because the project has no prior queries in production, this introduces no regression risk against existing behavior. However, developers must be aware that after this change, all `findMany`, `findFirst`, and `findUnique` queries on `Trip` and `User` models will have `deletedAt: null` injected automatically. Test coverage (unit + integration) must verify this behavior before merge.

**Verdict**: Non-Breaking. No migration plan required. No deprecation notice required. No external consumer notification required.

## Version Decision

### Semantic Version: 0.1.0

**Justification**: Per [SemVer 2.0.0](https://semver.org/), major version zero (`0.y.z`) is for initial development. The public API is not yet considered stable. Version `0.1.0` signals: the first minor increment of an unstable pre-1.0 product — a meaningful, functional feature set has been added, but the public API may still change without a major version bump.

`0.1.0` is preferred over `0.0.1` because this release delivers a complete, user-facing feature (not a trivial patch): it introduces a new data model, new Server Actions, new UI pages, new business logic, and a test suite. This warrants a MINOR increment within the `0.x.x` range.

### Version Strategy Going Forward

| Release | Trigger | Example |
|---------|---------|---------|
| Patch (`0.1.x`) | Bug fix in trip creation, form validation correction, cache key fix | `0.1.1` — fix endDate validation edge case |
| Minor (`0.x.0`) | New user-facing feature added backward-compatibly | `0.2.0` — add US-002 Itinerary Builder |
| Major (`1.0.0`) | Public API declared stable OR a breaking change is introduced to a previously stable contract | `1.0.0` — first production-ready release with stable REST API contract |

**Pre-1.0 breaking change policy**: Per SemVer, breaking changes during `0.x.x` do not require a MAJOR bump. However, this team will treat any change that requires a database migration of existing data, or that removes/renames an existing Server Action signature, as requiring at minimum a MINOR bump and a migration guide — regardless of the `0.x.x` pre-release status.

## Risk Assessment

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Database migration fails in production (e.g., foreign key constraint on `users` table missing) | Low | High | Run migration in staging environment first; verify `users` table exists before applying trips migration; maintain migration rollback script |
| Soft-delete middleware on `db.$use` is deprecated in Prisma 5+ (OQ-005 from SPEC-001) | Medium | Medium | Confirm Prisma 7 API: use `db.$extends` (Prisma Extensions) instead of `db.$use`; confirm with developer before implementation; add integration test verifying soft-delete filter is applied |
| `useActionState` signature mismatch with React 19 (OQ-004 from SPEC-001) | Low | Medium | Confirm Next.js 15 + React 19 `useActionState` API before implementing TripForm; reference official React 19 docs |
| Trip limit enforcement bypass via race condition (two concurrent createTrip calls for user at limit 19) | Low | Low | Count + insert is not atomic; for MVP the impact is a user having 21 trips instead of 20 — acceptable. Add database-level constraint or serializable transaction if needed post-MVP |
| Redis cache not invalidated on error path, serving stale data | Low | Medium | TripService cache invalidation wrapped in `Promise.all`; integration test covers cache invalidation after each mutation type |
| Performance: FCP > 1.5s on 4G (AC-015) | Low | Medium | Three composite indexes cover all query patterns; Next.js RSC + Vercel edge; measure with Lighthouse before release sign-off |
| `users` table not yet created (US-003 not yet merged) | Medium | High | CIA-001 deployment is blocked on US-003 schema migration being applied first; add explicit prerequisite check to deployment runbook |
| SPEC-001 status is "Draft" — not yet approved | Medium | High | This CIA is drafted against a spec still awaiting tech-lead and security-specialist approval; final CIA approval is contingent on SPEC-001 being marked Approved |

**Overall Risk Level**: Low-Medium — greenfield with no backward compatibility risk, but dependent on US-003 migration ordering and on resolution of SPEC-001 open questions (OQ-003, OQ-004, OQ-005).

## Rollback Plan

Because this is the first release and the system has no prior data, rollback is straightforward.

### Database rollback

```bash
# Revert the Prisma migration
npx prisma migrate resolve --rolled-back YYYYMMDDHHMMSS_create_trips_table

# Drop the trips table and enums manually if needed
DROP TABLE IF EXISTS "trips";
DROP TYPE IF EXISTS "TripStatus";
DROP TYPE IF EXISTS "TripVisibility";
```

### Application rollback

```bash
# Revert to the previous deployment (Vercel)
vercel rollback [deployment-url]

# Or via git: revert the feature branch merge commit and redeploy
git revert [merge-commit-sha]
git push origin main
```

### Redis rollback

No Redis data migration is performed by this feature. Cache keys created during operation will expire per their TTL (60s for list, 300s for detail) without manual intervention.

### Rollback decision trigger

Initiate rollback if any of the following occur within 30 minutes of deployment:
- Error rate on `/trips/*` pages exceeds 1% (measured via Sentry)
- Database migration fails or produces query plan regressions (P95 query time exceeds 200ms)
- Any 500 error on `createTrip`, `updateTrip`, `archiveTrip`, or `deleteTrip` Server Actions
- Soft-delete middleware leaking soft-deleted records in any query (verified via integration test suite)

## Release Notes Draft (v0.1.0 — for stakeholders, in Portuguese)

```
# Travel Planner — Versao 0.1.0
Data de lancamento: 2026-02-23

## O que ha de novo

Esta e a primeira versao do Travel Planner. Agora voce pode criar e
organizar suas viagens em um unico lugar.

## Funcionalidades disponiveis

**Criacao de viagens**
Crie uma nova viagem informando o nome, destino, data de partida e data
de retorno. Voce tambem pode adicionar uma descricao e escolher uma capa
personalizada com gradiente e emoji.

**Painel de viagens**
Acesse todas as suas viagens em `/trips`. As viagens sao exibidas em
ordem de data de partida (proximas primeiro) e podem ser filtradas por
status: Planejando, Ativas, Concluidas e Arquivadas.

**Edicao de viagens**
Edite qualquer informacao de uma viagem a qualquer momento. As alteracoes
sao salvas imediatamente.

**Arquivamento**
Viagens concluidas ou pausadas podem ser arquivadas. Elas saem da listagem
principal mas continuam acessiveis pelo filtro "Arquivadas".

**Exclusao**
Voce pode excluir uma viagem de forma permanente. Por seguranca, e
necessario digitar o nome da viagem para confirmar a exclusao.

## Limites do MVP

- Cada usuario pode ter ate 20 viagens ativas (status Planejando, Ativa
  ou Concluida) ao mesmo tempo. Viagens arquivadas nao contam para esse
  limite.
- Upload de imagem de capa sera disponibilizado em uma versao futura.
  Por enquanto, escolha entre 8 gradientes e um emoji personalizado.
- O campo de destino e texto livre nesta versao. Sugestao automatica de
  destinos via mapa sera adicionada em breve.

## Acessibilidade e dispositivos moveis

O formulario de criacao de viagem e totalmente operavel via teclado e
compativel com leitores de tela (WCAG 2.1 AA). A interface foi testada
em dispositivos moveis a partir de 375px de largura.
```
